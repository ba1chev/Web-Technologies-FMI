Криптографско API в уеб средата

Става дума за Web Cryptography API — начинът, по който браузърът предлага криптография на страницата (хешове, криптиране, подписи, работа с ключове), без да трябва да ползваме само чист JavaScript библиотеки. Източниците за фактите са на английски и са описани в края; самият текст тук е на български.

Въведение

В ежедневните приложения постоянно изникват задачи като вход, краткоживущи токени, криптиране на файл още в браузъра, или поне проверка, че данните не са пипани. Каналът до сървъра обикновено минава през TLS, но това не покрива всичко — понякога трябва операция изцяло при клиента. Преди да има стандарта, хората въртяха криптография в обикновен JS: става, но често е по-бавно, по-трудно за преглед и по-чувствително към грешки в употребата.

Web Cryptography API описва по-ниско ниво на достъп: браузърът върши тежката част през свой API, а ключът може да остане „заключен“ вътре — напр. като non-extractable, за да не излиза суровият материал без нужда [1]. В практиката го ползваме през window.crypto и crypto.subtle (тип SubtleCrypto) [2]. Думата subtle на практика значи: детайлите имат значение; объркаш ли режима или параметрите, схемата може да се счупи макар алгоритъмът да е „силен“ [2].

Какво гледаме по-надолу. Кратко за secure contexts, основните операции (хеш, HMAC, криптиране, подпис, ключове) и връзката със сървъра. Няма как да изброя поддръжката за всеки алгоритъм и всеки браузър — за това са таблиците при Mozilla, Can I use и т.н. [2],[6].

Типични сценарии, за които се ползва API-то

Не е задължително да ползвате всичко от първия ден. В реални проекти често се започва от един-два сценария и после се разширява. Ето няколко често срещани, които са в духа на самия стандарт [1]:

  • Проверка на цялост преди качване. Файлът се хешира в браузъра, качва се и на сървъра — после двата дайджеста се сравняват. Така хората виждат локално „отпечатъка“, преди да пратят големи данни [1],[2].
  • Криптиране още на устройството. Например клиентско криптиране преди облак: ключът не напуска машината на потребителя по начин, който приложението изрично не позволява [1].
  • Подпис на съдържание. Документ или пакет данни се подписва с частен ключ, а отсреща се проверява с публичен — класическият модел, но върху Web Crypto [1],[2].
  • Договаряне на сесия със сървър. Клиентът генерира или деривира ключ, предава само нужните публични части или опаковки, а TLS остава за канала [1],[7].

Във всеки случай си струва да се запише на хартия (или в дизайн документ): кой държи ключа, къде е солта, как се върти IV-то и какво точно гарантира сървърът. Иначе лесно се получава „криптирано“, но без ясен модел на заплахите [8].

Какво представлява Web Cryptography API

На ниво платформа нещата са подредени около обекта Crypto и „вътрешния“ интерфейс SubtleCrypto: генериране и деривация на ключове, подпис, проверка, криптиране/декриптиране, хеш върху ArrayBuffer [1],[2]. Не е нужно да помним всичко наизуст — важно е да се види кой слой какво прави; затова фиг. 1 е опростена картинка на стека, а фиг. 2 показва типичен ред на стъпките при повикване на subtle.

Извикванията са асинхронни и връщат Promise, за да не задръстват нишката на интерфейса при по-тежки сметки [2]. Стандартът също така държи нещата сравнително стегнати — не е „направи си криптото както ти харесва“, а списък от ясно именувани алгоритми и режими [1].

Какво има освен subtle

Същият глобален crypto обект носи и неща извън SubtleCrypto. Най-често се ползва crypto.getRandomValues, за да се напълни Uint8Array с криптографски подходящи случайни байтове [14]. Това не е мястото за Math.random() — той е детерминиран от по-стари генератори и не е подходящ за токени, соли или IV-та [14].

Импортът и експортът на ключове често минава през JWK (JSON) или бинарни формати като PKCS #8 / SPKI — удобно е при обмяна със сървър, но форматът трябва да съответства на очакванията на кода на гърба [1],[2].

Работниците (Web Workers) също могат да имат достъп до Web Crypto, което помага, ако искате тежки операции извън основната нишка. Детайлите зависят от типа работник и политиките на браузъра — проверката пак е експериментално с вашия билд [2].

Безопасен контекст и политики за достъп

crypto.subtle на практика изисква сигурен контекст (secure context) — най-често HTTPS или поне localhost, докато донагласяте проекта [3]. Ако това липсва, при различни браузъри или няма да видите subtle, или ще хвърля грешки; затова тестваме в реалната среда, а не само „на сухо“ [2],[3].

Идеята на модела е: чувствителни възможности (вкл. криптография в браузъра) да се пускат само когато контекстът е достатъчно сигурен, за да не се размива границата на платформената сигурност [3].

В код това често значи ранна проверка и ясно съобщение, ако сайтът е отворен по обикновен HTTP насред продукция. И mixed content може да обърка нещата — смесени нешифрирани ресурси към HTTPS страница са отделна тема, но засягат същите политики [3],[4].

Локално често отваряте referat.html през file://. При някои браузъри това не е сигурен контекст или subtle просто липсва — не че проектът е счупен, а че политиката е нарочно строга [2],[3],[13]. За демото по-долу и за собствени тестове е по-надеждно да пуснете малък локален сървър (например python3 -m http.server от папката на сайта и отваряне на http://127.0.0.1:…/referat.html).

SubtleCrypto: операции и ключове

През SubtleCrypto минават нещата като digest, sign, verify, encrypt, decrypt, деривация (deriveBits, deriveKey), импорт/експорт и generateKey [1],[2]. Флагът extractable още от пръв поглед показва дали някой се опитва да изкара суров ключ навън — удобно, когато целта е точно обратното [1].

Табл. 1 ги наредих по групи само за ориентир. Реалният „дали работи при мен“ списък пак е в документацията и таблиците за съвместимост [1],[2],[6].

Таблица 1. Групи методи на SubtleCrypto и типични сценарии.
Група Примерен метод Примерна употреба
Хеширане digest Цялост на файл, конструиране на key derivation вериги
Симетрично encrypt / decrypt AES-GCM, ChaCha20-Poly1305 (ако е налично)
Асиметрично sign / verify ECDSA, RSA-PSS
Ключове generateKey, deriveKey Създаване на сесийни ключове, HKDF, PBKDF2

Малък, но важен детайл: при AES-GCM IV-то не е украса — ако го преизползвате със същия ключ, режимът се счупва [5]. В клиентски код това се случва по-често, отколкото ни се иска, ако състоянието не се държи изрядно.

Алгоритми, съвместимост и ориентири за избор

Имената на алгоритмите не са произволен низ — има регистър: SHA-256, AES-GCM, PBKDF2, HKDF, ECDSA и т.н. Едно нещо е какво пише в спецификацията, друго е какво е включено в конкретния браузър [1]. В реален проект почти винаги има проверка „дали този import/generate минава“ [2],[6]. Няколко ориентира по често ползвани конструкции са дадени в табл. 2.

Таблица 2. Ориентировъчна поддръжка (индикативно; валидирайте в целевия браузър) [2],[6].
Алгоритъм / режим Бележка за поддръжка
SHA-256 (digest) Широко наличен в съвременни потребителски агенти
AES-GCM Чест избор за симетрично криптиране в браузъра при коректен IV
ECDSA с криви на NIST Подписи и удостоверяване на съобщения; валидирайте кривата и хеша [1],[5]
Ed25519 По-нова регистрация; сравнете таблиците за версиите на браузъра/двигателя [1],[2],[12]

Ако някой реши да „хешира паролата още в браузъра“, само digest не става — трябва нещо бавно и солено като PBKDF2 (или еквивалент със смислена цена на операцията), плюс какво точно прави сървърът после [9]. И да — това не отменя нуждата от TLS по мрежата [4],[7].

Подводни камъни (накратко)

Три неща, които виждам често при курсови и малки прототипи: (1) да се ползва един и същ IV при GCM „за по-лесно тестване“; (2) да се хешира парола само с digest и да се вярва, че това е „солена“ защита; (3) да се пренебрегне сигурният контекст и после да се чудим защо subtle е undefined [2],[5],[9]. Разделете ясно: какво защитава TLS, какво прави клиентската криптография и какво вярва сървърът след получаване на данните [7],[8]. За бързо сравнение „чиста JS библиотека“ срещу нативния Web Crypto виж табл. 3.

Таблица 3. Сравнение на подход: чист JS библиотека срещу нативен Web Crypto (обобщено) [1],[2].
Критерий Библиотека в JS Web Cryptography API
Скорост Зависи от JS двигателя; тежките режими усещат се на слаби устройства. Обикновено по-близо до нативен код на платформата [2].
Управление на ключове Ключовете са байтове в паметта на страницата — лесно за дебъг, трудно да се скрият. Има път чрез non-extractable ключове и импорт/експорт по правила [1].
Одит и повършина Повече код под ваш контрол — и повече място за грешка. По-малък „периметър“ в приложния JS, но параметрите пак трябва да са верни [1],[2].
Офлайн / среда Работи и там, където няма subtle (но тогава сигурността е друг разговор). Иска сигурен контекст и съвместим браузър [2],[3].

Интеграция с екосистемата: TLS, сървър и бъдещи разширения

TLS пази тръбата до сървъра [7]. Web Crypto е за неща вътре в страницата: подпис преди изпращане, криптиране на blob, деривиране на ключ преди мрежовата заявка и т.н. Как точно ги комбинираш — вече зависи от модела на заплахите [4],[8].

За сравнение, WebAuthn е друг стандарт за силен вход с ключове и автентикатори [10] — тема със собствени детайли, но в същия ъгъл на сигурността. За случайни байтове пък обикновено ползваме crypto.getRandomValues [14].

Накрая, стандартът не премахва нуждата от добра сървърна логика: подписът или ciphertext-ът трябва да се приемат с ясни правила, да има версиониране на протокола и да не се разчита единствено на „невидимост“ на клиентския код [8].

Малко интерактивно демо

Това не е пълноправно приложение, а учебен експеримент в самата страница: вижда се дали средата отговаря на изискванията за crypto.subtle и може да се пресметне един SHA-256 дайджест от произволен текст. Стойностите идват директно от браузъра ви [2],[13].

Протокол
{{ env.protocol }}
isSecureContext
{{ env.secure ? 'да' : 'не' }}

Тук subtle не е достъпен — демо бутонът няма да работи. Пуснете страницата през http://localhost или друг сигурен контекст [2],[3],[13].

Илюстративни програмни фрагменти

Две мини-примера с async/await (може и чисти Promise, ако предпочитате) [2]. Код 1 — SHA-256; код 2 — ключ и AES-GCM.

Код 1. Хеширане чрез digest (SHA-256)

async function sha256Hex(messageUtf8) {
  const enc = new TextEncoder();
  const data = enc.encode(messageUtf8);
  const digest = await crypto.subtle.digest('SHA-256', data);
  const bytes = new Uint8Array(digest);
  return Array.from(bytes, (b) => b.toString(16).padStart(2, '0')).join('');
}

// Пример: await sha256Hex('hello web crypto');

Код 2. Генериране на AES-GCM ключ и криптиране

async function aesGcmEncrypt(plaintextUtf8) {
  const iv = crypto.getRandomValues(new Uint8Array(12)); // 96-битов IV за GCM
  const enc = new TextEncoder();
  const plaintext = enc.encode(plaintextUtf8);

  const key = await crypto.subtle.generateKey(
    { name: 'AES-GCM', length: 256 },
    false, // неекстрактируем ключ
    ['encrypt', 'decrypt']
  );

  const ciphertext = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv },
    key,
    plaintext
  );

  return { iv, ciphertext }; // ciphertext е ArrayBuffer
}

Няма да тръгнат без subtle и без сигурен контекст; в „истински“ проект добавяме още проверки за грешки и ясно кога ключът живее [1],[2],[5].
